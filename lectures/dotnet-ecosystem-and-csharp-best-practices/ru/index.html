Обзор .NET платформы

Сложность: Easy peasy lemon squeezy. Цель: Почитать/послушать о платформе .NET в целом (много времени не займет).

Сейчас компания Microsoft вкладывает достаточно много ресурсов в развитие .NET и их облачной платформы — Azure (Ажур, Эйжур, Ейжа) и больше всех контрибьютит в open source. Инфраструктура .NET стремительно растет и развивается, постоянно создаются новые инструменты для работы с ней, добавляются интеграции с другими сервисами.

    Пути создания .NET-приложений:

    1. .NET Framework — разработка под Windows настольных приложений на Windows Forms, WPF, веб-серверов на ASP.NET и WCF
    2. .NET Core — разработка кроссплатформенных веб-приложений с помощью ASP.NET Core, создание гибридных приложений с помощью Universal Windows Platform, которая позволяет запускать программу, написанную на этой технологии, на Windows-машине, Xbox, Hololens
    3. Xamarin — платформа для создания мобильных приложений для iOS и Android, используя C#, XML и XAML

    Код, написанный под специальный фреймворк, такой как WPF, ASP.NET Core или Android, нельзя переиспользовать на другой платформе, потому что он заточен для работы с так называемыми platform-specific API, который отличается у всех. Чтобы можно было повторно использовать код бизнес-логики, хелпер-методов, моделей, классов и т.д. был создан .NET Standard. Он предоставляет набор доступных АPI, которые одинаково работают во всех десктопных программах, веб-серверах, мобильных приложениях, играх и облачных службах вне зависимости от операционной системы и платформы.

  • В ноябре 2020 года компания Microsoft выпустила платформу .NET 5, которая стала серьезным улучшением системы разработки в целом. Главным нововведением по сравнению с .NET Framework 4.8, стало обеспечение поддержки Linux, macOS, iOS, Android, tvOS, watchOS и WebAssembly. При этом версия Framework 4.8 стала последней, дальше развиваться будет только семейство .NET x. Это позволило унифицировать платформу и предложить единственный открытый фреймворк и runtime для различных задач. В результате можно создавать приложения для различных платформ на общей кодовой базе с одинаковым процессом сборки независимо от типа программы.     А уже на ноябрь 2021 года запланирован релиз .NET 6.

  • Каждому программисту рано или поздно приходится имплементировать функционал, который частично или полностью кто-то уже создал и даже опубликовал (как правило в виде DLL библиотеки). Разработчики называют такие модули «пакетами», в которые собран скомпилированный код, дополнительные файлы-ассеты и манифест, объясняющий цель и способ использования пакета. В большинстве языков программирования есть собственные платформы для обмена такими полезными модулями. В .NET это NuGet, поддерживаемый Microsoft-ом. Разработчики, которые создали крутой инструмент или, например, библиотеку для работы с файловой системой, могут опубликовать свою работу как NuGet-пакет в виде zip-файла з расширением .nupkg. Вы можете искать и скачивать модули, которые ускорят разработку вашего приложения, из центрального репозитория NuGet Gallery ― он насчитывает уже около 250 000 уникальных пакетов и там может найтись что-то полезное.

.NET под капотом

Сложность: Hard as hell 🔥 Цель: Понять SDK.

  • Сегодня в .NET-мире самые распространенные языки программирования - это C#, F# и Visual Basic. У каждого есть свой компилятор, который преобразует код, написанный на этом языке, в Intermediate Language Code (IL). Последний представляет собой набор инструкций для виртуальной машины .NET — CLR (Common Language Runtime).

    Основные этапы выполнения программы .NET:

    • Обычный C# код
      public void SumTwoNumbers() {
  int firstNumber = 10;
  var secondNumber = 200;

  Console.WriteLine(firstNumber + secondNumber);
}
      C# код, скомпилированный в IL
      .method public hidebysig

  // Method begins at RVA 0x2070
  // Code size 18 (0x12)
  .maxstack 2.locals init([0] int32 firstNumber, [1] int32 secondNumber)

  IL_0000: ldc.i4.s 10
  IL_0002: stloc.0
  IL_0003: ldc.i4 200
  IL_0008: stloc.1
  IL_0009: ldloc.0
  IL_000a: ldloc.1
  IL_000b: add
  IL_000c: call void[mscorlib] System.Console::WriteLine(int32)

}
    • А когда придет время для выполнения части кода в программе, CLR с помощью JIT (Just in Time) компилятора превратит код IL в машинный код.

    Результатом билда .NET-программы является файл с расширением .exe (Executable) или .dll (Dynamic Link Library).

    Важно отметить, что при преобразовании IL в машинный код будет преобразована только та часть кода, которая должна выполниться в текущий момент времени.

  • На высшем уровне в C# есть 2 типа данных — это значимые типы (value types) и ссылочные типы (reference types). Важно понимать различия между ними:

    Значимые типы:
    • Целочисленные типы
    • Типы с плавающей точкой
    • decimal
    • bool
    • enum-ы
    • структуры
    Ссылочные типы:
    • тип object
    • string
    • классы
    • интерфейсы
    • делегаты

    Значимые типы хранятся в стеке, ссылочные — в куче. Value types передаются по значению, то есть копируются, reference types передаются по ссылке.

  • В .NET память делится на два типа: стек и кучу. Стек представляет собой структуру данных, которая растет снизу вверх: каждый новый элемент размещается поверх предыдущего. В стеке хранятся значимые типы и ссылки на ссылочные типы, которые, в свою очередь хранятся в куче.

    Кучу можно представить как неупорядоченный набор разнородных объектов. При создании объекта ссылочного типа в стек добавляется ссылка на адрес этого объекта в куче. Когда объект cсылочного типа перестает использоваться, то ссылка удаляется из стека, и память освобождается.

    В .NET-е очистка памяти происходит автоматически. За это отвечает Garbage Collector (по нашему — сборщик мусора). Когда он видит, что на объект в куче больше нет ссылок, он удаляет этот объект и очищает память.

  • Важный момент в том, как переменные значимых и ссылочных типов передаются в метод.

    Value Type

    static void Main(string[] args) {
  int c = 20;

  MethodValue(c);

  Console.WriteLine(c); // 20
}

// pass copy of the value
static void MethodValue(int variableCopy) {
  variableCopy = 1;
}

    Reference Type

    class City {
  public int code;
  public City(int code) {
    this.code = code;
  }
}

static void Main(string[] args) {
  City city = new City(12);

  MethodReference(city);

  Console.WriteLine(city.code); // 0
}

// pass the reference to the object on heap
static void MethodReference(City city) {
  city.code = 0;
}

    Ref

    static void Main(string[] args) {
  int d = 30;

  MethodValueRef(ref d);

  Console.WriteLine(d); // 2
}

// pass value by reference 
static void MethodValueRef(ref int variable) {
  variable = 2;
}

    Out

    static void Main(string[] args) {
  int e;

  MethodValueOut(out e);

  Console.WriteLine(e); // 3
}

// pass value by reference
static void MethodValueOut(out int variable) {
  variable = 3;
}

  • Операция упаковки — boxing — это выделение памяти в куче под объект значимого типа — value type, и присваивание ссылки на эту область памяти переменной в стеке. Распаковка — unboxing, наоборот выделяет память в стеке под объект, полученный из кучи по ссылке.

    int i = 123;      // a value type
object o = i;     // boxing
int j = (int)o;   // unboxing
Раздел 3 наверх

Самые вежные темы C#

Сложность: Ну так себе. Цель: Понять SDK.

  • Структуры по виду очень похожи на Классы, но существует принципиальное отличие, которое упоминалась ранее. Класс — это reference type и передается по ссылке, а структура — value type и передается по значению — то есть, копируется.

    Структуры лучше использовать для небольших классов, маленьких структур данных и нетяжелых объектов. Классы же можно использовать во всех случаях, где вам неудобно использовать структуру. Они отлично подходят для того, чтобы быть частью иерархии сущностей, иметь внутреннее состояние и содержать в себе большое количество бизнес логики.

  • Классы и структуры могут иметь статические поля, методы и свойства. Если член статический, то он относится ко всему классу или структуре и для обращения к нему не нужно создавать экземпляр.

    class Wallet
{
    public static int Bonus = 100;
    public int balance;
    public Wallet(int bal)
    {
        balance = bal; 
    }

    public int GetBalance()
    {
        return balance + Bonus;
    }

    public static int GetPureBalance()
    {
        // Error: Cannot access non-static field in static context
        return balance;
    }
}

static void Main(string[] args)
{
    Console.WriteLine(Wallet.Bonus);      // 100
    Wallet.Bonus += 200;
     
    var wallet1 = new Wallet(150);
    Console.WriteLine(wallet1.GetBalance());   // 450
}

    На примере показано, что статическое поле является общим для всех объектов класса и может использоваться в нестатических методах. В то же время в статических методах у нас нет доступа к нестатическим членам класса.

  • Используя ключевое слово params мы можем «сказать», что наш метод принимает неопределенное количество параметров — это может быть ноль или больше, любое количество.

    public class Program
{
    public static void UseParams(string str, params int[] list)
    {
        Console.Write(str);
        for (int i = 0; i < list.Length; i++)
        {
            Console.Write(list[i] + " ");
        }
    }

    static void Main()
    {
        // You can send a comma-separated list of arguments
        // of the specified type.
        UseParams("params: ", 1, 2, 3, 4);
        // params: 
        // 1 2 3 4
    }
}

    Когда у метода переменное количество параметров, мы передаем аргументы ему просто перечисляя их через запятую, как показано на примере. Стоит отметить, что аргумент params должен быть указан последним, после перечня всех строго определенных аргументов метода.

  • В C# абстракция используется для сокрытия деталей реализации. Это значит, что мы сосредотачиваемся на том, что объект может делать, а не на том, как он это делает. Это часто используется при написании больших и сложных программ. Основные инструменты для этого — абстрактные классы и интерфейсы.

    В абстрактном классе мы можем создать функционал, который реализуется в унаследованном от него классе.. Со своей стороны интерфейс позволяет определить функциональные возможности или функции, но не может их реализовать..

    Класс имплементирует интерфейс и обязательно реализует эти методы. Рассмотрим несколько ключевых различий между ними:

    1. У интерфейса не может быть модификаторов доступа к членам — все что есть в интерфейсе является публичным по дефолту. У абстрактного класса все остается, как и у обычного класса.
      interface TestInterface  
{  
    // Causes syntax error
    protected void GetMethod();  

    public string PublicProp { get; set; } 
}
      abstract class TestAbstractClass  
{ 
    public abstract string GetStuff();

    public abstract void DoSmth();
}
    2. В интерфейсе мы можем только описать сигнатуру метода без его имплементации. А в абстрактном классе могут находиться как абстрактные методы и свойства, так и не абстрактные — с полной или частичной реализацией.
      interface TestInterface  
{  
    // Only signature
    void GetMethod();  
}



  
      abstract class TestAbstractClass  
{ 
    // Complete method implementation
    public string GetStuff()
    {
        Console.WriteLine("Stuff");
        return "Stuff";
    }  
}
    3. Мы не можем объявить конструктор в теле интерфейса ни с модификатором доступа, ни без него. В абстрактном классе мы можем объявлять конструкторы по тем же правилам, что и в обычных классах. Он в основном используется для вызова в конструкторе производного класса, чтобы не дублировать код инициализации полей или свойств абстрактного класса.
      interface TestInterface
{
    // Interfaces cannot contain instance constructors
    TestInterface()
    {
    }

    public TestInterface(int s)
    {
    }
}
      abstract class TestAbstractClass  
{  
    public int a;  
    public string b;

    // Everything is OK - we can do that
    public TestAbstractClass(int a, string b)  
    {  
        this.a = a;
        this.b = b;
    }  
}
    4. Мы не можем явно создать инстанс интерфейса или абстрактного класса вызвав конструктор. Хотя напомню, что у абстрактного класса он может быть.
      static void Main(string[] args)
{
    // Causes syntax error
    TestInterface testInterface = TestInterface();

    // Causes syntax error as well
    TestAbstractClass abstractClass = TestAbstractClass();
}
    5. У абстрактного класса могут быть поля и свойства, у интерфейса — только свойства.
      interface TestInterface  
{  
    // Causes syntax error
    int field;

    const string name;
}
      abstract class TestAbstractClass  
{ 
    // 
    public static int n = 1;
    protected const int m = 3;
    private int k = 3;  
}
    What, why and where?

    Интерфейс мы используем для описания API для нескольких классов, которые, скорее всего, будут реализовывать более чем один интерфейс. Надо помнить, что члены интерфейса не могут быть статическими. С#, в отличие от С++, не поддерживает множественное наследование, поэтому чтобы его реализовать, мы используем интерфейсы.

    Абстрактный класс используется, если мы хотим включить его в иерархию наследований и создать функционал с полной или частичной реализацией, которую класс-наследник может имплементировать или переопределить. Абстрактный класс позволяет сохранять состояние класса в целом, а не только отдельного его объекта.

    Интерфейс в основном используется тогда, когда мы хотим просто описать API использования классов, которые будут имплементировать этот интерфейс — задать поведение.

  • IDisposable объявляет единственный метод Dispose, в котором при имплементации интерфейса в классе должно происходить освобождение неуправляемых ресурсов, таких как соединения с базой данных, файловые дескрипторы, сетевые подключения и тому подобное. Неуправляемые ресурсы нужно освобождать как можно быстрее, еще до удаления объекта из памяти когда до него доберется Garbage Collector. Например, наш класс взаимодействует с файловой системой — открывает файл, читает что-то из него, записывает. И лучше как можно раньше закончить работу с этим файлом, чтобы другие программы или потоки могли его использовать. И еще такое, нам самим нужно явно вызывать метод Dispose, потому что Garbage Collector ничего о нем не знает. Это лучше всего делать в блоке try...finally, чтобы даже если возникнет ошибка, мы смогли освободить ресурсы и правильно очистить память.

    • class MyFile : IDisposable
{
    private FileStream MyFileStream { get; set; }
    private StreamWriter MyStream { get; set; }

    public MyFile(string filePath)
    {
        MyFileStream = File.Create(filePath);
        // Open stream for working 
        MyStream = new StreamWriter(MyFileStream);
    }

    public void AddTextToFile(string text) => MyStream?.Write(text);

    // Free unmanaged resources
    public void Dispose()
    {
        MyStream?.Dispose();
        MyFileStream?.Dispose();
    }
}
  
static void Main(string[] args) 
{  
    MyFile file = null;
    try 
    {
        file = new MyFile("D://file.txt")
    }
    finally 
    {
        file.Dispose();
    }

    // The same - but shorter way
    using (MyFile file = new MyFile("D://file2.txt")) 
    {  
        file.AddTextToFile("Hello"); 
    }
}

  • Extension методы позволяют «добавлять» методы в существующие типы без создания нового производного типа, перекомпиляции или модификации оригинального типа.Extension метод это особый статический метод, который обязательно должен быть членом статического класса.

    public static class StringExtensions
{
    public static int WordCount(this string str)
    {
        return str.Split(new char[] { ' ', '.', '?' },
                         StringSplitOptions.RemoveEmptyEntries).Length;
    }
}

string text = "Something like that!";  
int amountOfWords = text.WordCount(); // 3

    На примере показан Extension метод для типа String. Статический класс может иметь произвольное название, в то время как название метода должно отличаться от уже существующих методов в классе, который мы расширяем, или иметь другую сигнатуру. В дальнейшем мы можем использовать объявленный нами метод так же как и обычные методы класса, который мы расширяем.

  • Дженерики появились в C# 2.0. Они привнесли в .NET концепт типизированных параметров ― это позволяет проектировать классы и методы, которые определяют тип членов класса или метода только при инициализации.

    Например, используя общий параметр типа Т,мы можем написать единственный класс, который будет использоваться клиентским кодом без риска осуществления boxing-операций (которые сами по себе являются тяжелыми операциями, и злоупотреблять ими не есть хорошо).

    • class MyGenericClass<T>
{
    private T genericMemberVariable;
    public MyGenericClass(T value)
    {
        genericMemberVariable = value;
    }

    public T genericProperty { get; set; }

    public T GenericMethod(T genericParameter)
    {
        Console.WriteLine("Parameter type: {0}, value: {1}", 
                          typeof(T).ToString(),genericParameter);
        Console.WriteLine("Return type: {0}, value: {1}", 
                          typeof(T).ToString(), genericMemberVariable);

        return genericMemberVariable;
    }
}

      Как видно по приведенному выше коду, MyGenericClass определен с <T>. <T> указывает на то, что MyGenericClass является дженериком, и тип Т будет определен позже. Вы можете использовать любые буквы или слово вместо T, это не имеет значения.

    • class TestGenericClass
{
    static void Main()
    {
        // Declare an object of type int.
        var intGenericClass = new MyGenericClass<int>(10);

        int val = intGenericClass.GenericMethod(100);    
        /* Output:
            Parameter type: int, value: 100 
            Return type: int, value: 10
        */
    }
}

  • Делегаты — это объекты, которые указывают на методы; с помощью них мы можем вызывать методы, присвоенные делегату. Делегаты позволяют представлять методы в виде объектов и передавать их в функции, использовать как колбеки.

    delegate void LogMessage(string messageToLog);

    public class Program
    {
        static void LogRedMessage(string message)
        {
            Console.ForegroundColor = ConsoleColor.Red;
            Console.WriteLine(message);
        }
        
        static void LogGreenMessage(string message)
        {
            Console.ForegroundColor = ConsoleColor.Green;
            Console.WriteLine(message);
        }

        static void Main(string[] args)
        {
            // Create delegate variable and assign method's address to it
            LogMessage logMessage = LogRedMessage;
            
            // Invoke method assigned to this delegate(Invoke syntax)
            logMessage.Invoke("Hello world!\n");

            // Add another method to delegate's invocation list
            logMessage += LogGreenMessage;
            
            // Invoke all methods assigned to this delegate
            logMessage("Second message");

            // Remove the method from the delegate
            logMessage -= LogGreenMessage;
        }
    }
}

  • В C# Есть массивы, которые хранят в себе наборы однотипных объектов, но работать с ними не всегда удобно. Так как массив хранит фиксированное количество объектов, в случаях, когда мы заранее не знаем, сколько у нас их будет, гораздо удобнее будет применять коллекции.

    При выборе коллекций определяющую роль может сыграть то, что некоторые из них реализуют стандартные структуры данных, такие как:

    • стек
    • очередь
    • словарь
    • хэш-таблица

    ...которые могут пригодиться для решения различных специальных задач. Основой для создания всех коллекций является реализация интерфейсов IEnumerator и IEnumerable.

    Интерфейс IEnumerator представляет перечислитель, который делает возможным последовательный перебор коллекции, например в цикле foreach, или средствами LINQ. А интерфейс IEnumerable через свой метод GetEnumerator предоставляет перечислитель всем классам, которые реализуют данный интерфейс. Поэтому интерфейс IEnumerable является базовым для всех коллекций.

    Конкретные методы и способы использования могут отличаться от одного класса коллекции к другому, но общие принципы будут одними и теми же для всех классов коллекций.

    • В приведенном примере используются две коллекции: non-generic — ArrayList, и generic — List. Сейчас хорошей практикой считается использовать дженерик версии коллекций везде, где это только возможно — из-за строгой типизации и удобства в использовании. Большинство коллекций поддерживают добавление элементов.

      class Program
{
    static void Main(string[] args)
    {
        // non-generic collection ArrayList
        ArrayList objectList = new ArrayList() { 0, "someString", 'c', 19.0d };
        
        // generic collection - List
        List<string> countries = new List<string>() { "Lviv", "Kyiv", "Odessa", "Dnipro" };

        object obj = 12.3;
        objectList.Add(obj);

        // Error - because object is not a string type
        countries.Add(obj);

        objectList.RemoveAt(0); // delete element with index 0 - first element of collection
        foreach (object o in objectList)
        {
            Console.WriteLine(o);
        }
        Console.WriteLine($"Total amount of Collection's elements: {objectList.Count}");
    }
}

      Например, в данном случае добавление производится с помощью метода Add, но для других коллекций название метода может отличаться. Также большинство коллекций реализуют удаление (в данном примере производится с помощью метода RemoveAt, который удаляет элемент из коллекции по индексу).     • Stack<T> представляет коллекцию, которая использует алгоритм LIFO — last in — first out — ("последним пришел - первым вышел"). При такой организации данных каждый следующий элемент помещается поверх предыдущего. Получение элементов из коллекции происходит в обратном порядке — извлекается тот элемент, который находится выше всех в стеке.

      В классе Stack можно выделить два основных метода, которые позволяют управлять элементами - это:

      • Push: добавляет элемент в стек на первое место
      • Pop: получает первый элемент из стека
      • Peek:* просто возвращает первый элемент из стека без его удаления
      static void Main(string[] args)
{
    var stackCities = new Stack<string>();
    stackCities.Push("Lviv");
    stackCities.Push("Kyiv");
    stackCities.Push("Odessa");
    
    var odessa = stackCities.Pop();
    
    Console.WriteLine("Deleted city: " + odessa)

    foreach(string city in stackCities) {
        Console.WriteLine(city);
    }
    /* Output:
        Deleted city: Odessa
        Lviv
        Kyiv
    */
}
      На примере можно увидеть как мы создаем экземпляр коллекции стека лент. Добавляем 3 города — "Lviv", "Kyiv", "Odessa" с помощью метода Push. Вытаскиваем элемент который мы добавили последним с помощью метода Pop и выводим результаты на экран.

  • Иногда при выполнении программы возникают ошибки, которые трудно или невозможно предусмотреть (например, при передаче файла по сети может оборваться подключение пропасть интернет). Такие ситуации называются Exception-ами. Язык C# предоставляет разработчикам возможности для обработки таких ситуаций с помощью конструкции try...catch...finally

    • static void Main(string[] args)
{
    StreamReader file = new StreamReader(@"D:\test.txt");
    char[] buffer = new char[10];
    try
    {
        file.ReadBlock(buffer, 123, buffer.Length);
    }
    catch (Exception e)
    {
        Console.WriteLine(e.Message);
    }
    finally
    {
        file?.Close();
    }
}
      1. При использовании блока try...catch...finally сначала выполняются все инструкции в блоке try.
      2. Если в этом блоке не возникло Exception-ов, то после него исполнится блок finally и конструкция try..catch..finally завершит свою работу.
      3. Если же в блоке try возникает Exception, то обычный поток выполнения останавливается, и CLR начинает искать блок catch, который может обработать этот Exception.
      4. Если блок catch найден, то он выполняется, а после его завершения исполнится блок finally.
      5. Если нужный блок catch не найден, то программа аварийно завершает свое выполнение.

    • В C # все типы Exception-ов наследуются от родительского класса Exception, который дополнительно делится на две ветви SystemException и ApplicationException.

Раздел 4 Наверх

Принципы чистого кода

Сложность: Семь раз отмерь, один раз отрежь. Цель: Понять, как писать такой код, который захочется читать.

  • В переводе на человеческий язык — общепринятые стандарты написания кода и согласованные правила, как называть переменные, функции и др. Это — грамматика и орфография C#, принятая большинством .NET-разработчиков для того, чтобы другие девелоперы (вы через Х времени) могли легко и быстро понять, что происходит в вашем коде и использовать его, не путаясь во всех возможных способах назвать, скажем, аргумент (и такие правила написания есть абсолютно во всех языках программирования, не только в C#). Долго рассказывать о каждом из правил нет смысла, главное для вас — самостоятельно ознакомиться со списком большинства распространенных стандартов C#.

    Хотите быть цивилизованным разработчиком и заслужить уважение среди коллег — прочтите несколько абзацев с примерами правильно оформленного кода и придерживайтесь такого формата, выполняя таски малые или большие.

  • Не повторяйся при написании кода = не прописывай несколько раз то, что можно закодить один раз, и вызывать, обращаясь к конкретному модулю. Пример — веб-приложение, содержащее несколько одинаковых по оформлению блоков, где каждый имеет собственное (ідентичний іншим!) описание стилей. Какова вероятность, что когда нужно будет сделать одинаковую правку во всех этих блоках (вручную, ведь мы несколько раз повторяем тот же набор стилей), разработчик пропустит один или несколько из них? Когда этот принцип нарушен и имплементация метода или даже класса дублируется без настоящей потребности, а написано несколько сотен тысяч строк кода (как на любом реальном проекте), то для рефакторинга, изменения бизнес-логики или даже простых изменений в интерфейсе, приходится долго искать по имени метода несчастный кусок кода, часто для того, чтобы поменять в нем только 1 цифру.

    Короче, так не делается 😐. Чтобы достичь DRY в вашем коде — делите его на маленькие кусочки; видите что часть логики повторяется — сразу выносите, объединяйте функции. Почему нужен DRY? Чем меньше кода, тем лучше. Его легче поддерживать, меньше времени уходит на то, чтобы в нем разобраться, и также уменьшается количество багов.

  • Этот принцип говорит сам за себя — простой и лаконичный код легче понять другим разработчикам и тебе, когда ты вернешься к нему через некоторое время. Он формулируется так — «каждый метод должен решать только одну маленькую проблему, а не иметь много различных средств потребления». Если в методе много условий, то разбейте их на меньшие методы. Такой код легче читать, поддерживать, и это также помогает находить ошибки гораздо быстрее. Чтобы продемонстрировать KISS, чаще всего приводят пример с определением дня недели:

    Simple
    string weekday(int day) {
  switch (day) {
  case 1:
    return "Monday";
  case 2:
    return "Tuesday";
  case 3:
    return "Wednesday";
  case 4:
    return "Thursday";
  case 5:
    return "Friday";
  case 6:
    return "Saturday";
  case 7:
    return "Sunday";
  default:
    throw new Exception("Day must be in range 1 to 7");
  }
}
    Stupid
    string weekday(int day) {
  if ((day < 1) || (day > 7)) {
    throw new Exception("Day must be in range 1 to 7");
  }

  string[] days = {
    "Monday",
    "Tuesday",
    "Wednesday",
    "Thursday",
    "Friday",
    "Saturday",
    "Sunday"
  };

  return days[day - 1];
}
    На слайде показаны два метода для решения этой задачи:
    1. Первое решение простое как двери — простой switch с дефолтным case-ом в случае если день не найдены.
    2. Второй метод тоже рабочий, но для того чтобы его понять, нужно дольше вчитываться
    Такой код существует повсюду, но он действительно неудобный и выглядит непрофессионально, 99% программистов выбрали бы работать с чем-то похожим на первый вариант.
    Чтобы достичь KISS ― старайтесь писать максимально простой код. Если видите сложный (нечитаемый) участок кода, поищите более лаконичное решение той же самой задачи, и, отрефакторив написанное, вы удивитесь, что кусок на 200 строк на самом деле не так уж и необходим!

  • SOLID — это 5 принципов объектно-ориентированного программирования, которые описывают архитектуру программного обеспечения:

    А если проще, то это правила, следуя которым, вы будете писать легкий для понимания, редактирования или повторного использования код.

    • Принцип единой ответственности. Он означает, что каждый класс или структура должны иметь только одну задачу.. Все члены класса пишутся для выполнения данной ему задачи, и в нем не находится ни строчки кода, которая не относится к указанной для этого блока задачи. Если мы придерживаемся этого принципа, то мы определяем классы по их задачам еще на этапе проектирования программы.

      😢

      class Task {
  public string Title { get; set; }
  
  public string Description { get; set; }

  // Adds task in Database
  public bool Add(Task tast) {
    // Internal realization (Insert into DB)
  }

  // Estimate Task's duration
  public void EstimateTaskDuration(Task tast) {

  }

      class Task {
  public string Title { get; set; }
  
  public string Description { get; set; }

  // Adds task in Database
  public bool Add(Task tast) {
    // Internal realization (Insert into DB)
  }

      Я привел в пример класс Task — он сохраняет задачу в нашу базу данных и вычисляет время, необходимое для решения задачи.
      Делаем вывод, что он не соответствует принципу Single Responsibility Principle. Почему мы не хотим, чтобы он выполнял и другие полезные функции, например, определял необходимое для выполнения задачи время? Потому что если через некоторое время у заказчика меняются параметры выполнения задач (например, из-за релиза или изменения в составе команды программистов), нам придется переписывать класс Task в соответствии с изменениями в исходных данных, и тестировать, не сломался ли при том иной функционал, который Task выполняет. Согласно Single Responsibility Principle мы должны создать отдельный класс для расчета времени на выполнение заданий, который уже будет руководствоваться бизнес-логикой и другими входящими данными.

    • Принцип открытости / закрытости. Наш класс должен быть открытым для масштабирования, но закрытым для модификаций. Наш модуль должен быть разработан так, чтобы дописывался он только при создании новых требований — но касающихся начальной задачи. «Закрытый для модификаций» означает, что класс уже полностью готов и жизнеспособен, его задачи и назначения не меняются, следовательно мы не переписываем его существенно, кроме как в случае исправления багов. В C# это достигается принципом наследования.

      Рассмотрим пример с мокапом — прототипом страницы. Проблема данного класса в том, что когда заказчик хочет посмотреть на созданный дизайнерами мокап, но не может открыть изображение формата tiff или cdr, разработчику нужно будет вносить новый формат картинки, например png. Поэтому мы будем вынуждены добавить новое условие if, что противоречит Open Closed Principle.

      На втором примере показано как это можно решить — есть базовый абстрактный класс Mockup, который частично реализует конвертацию картинки, а дочерние классы реализуют конвертацию картинки в необходимый формат. И если мы захотим добавить еще один формат, нам просто надо будет создать еще один класс, который наследуется от Mockup и реализовывает нужный нам метод конвертации.

    • По принципу подстановки Лисков мы должны использовать любой дочерний класс вместо родительского таким же образом, не внося изменений. Дочерний класс не может нарушать определение типа, приведенные в родительском классе и противоречить его поведению собственным функционалом.

  }

  // C# Developer can't  create Front-End App

      interface IFrontend {
  string CodeWebApp();
}

interface IBackend {
  string CodeServer();
}

class JavaScriptDeveloper: IFrontend,
IBackend {
  public string CodeWebApp() {
    return "Coding Front-End with Angular";
  }
  public string CodeServer() {
    return "Coding Back-End with Node.js";
  }
}

class CSharpDeveloper: IBackend {
  public string CodeServer() {
    return "Coding Back-End with ASP.Net";
  }
}
      Вот как это можно проиллюстрировать: Developer является родительским классом для JavaScriptDeveloper и CSharpDeveloper. Наш класс Developer может создавать бэкенд и фронтенд приложения. Казалось бы все хорошо. JavaScriptDeveloper успешно реализует 2 метода. А вот с CSharpDeveloper не все так просто, он может написать сервер на ASP.NET-е, но совершенно не может во фронтенд. И если мы попробуем все же получить от него фронтенд, то поймаем ошибку - exception. По-хорошему нам нужно разделить функционал Developer на 2 части: IFrontend и IBackend, и реализовать их в соответствии с назначением производных классов: JavaScriptDeveloper у нас реализует и IFrontend и IBackend, а CSharpDeveloper только IBackend.

    • Принцип разделения интерфейсов гласит, что не надо паковать вместе все интерфейсы подряд, надо их разделять по назначениям, чтобы пользователи могли выборочно имплементировать только те, которые используют, а не все подряд, имеющиеся в программе.

  string CodeServer();
}

      interface IDesktop {
  string CodeDesktop();
}

interface IBackend {
  string CodeServer();
}

      Давайте предположим, что у нас есть интерфейс IDeveloper, который теперь умеет создавать сервер и десктопное приложение. Как и раньше, у нас есть JavaScriptDeveloper и CSharpDeveloper, которые могут использовать этот функционал по назначению. Для JavaScript приложение писалось бы под Electron, а на C# это было бы WPF приложение. Все классно, все довольны, но нет, потому что наш босс неожиданно говорит, что приложения на Electron у него лагают и вообще дорого платить этим JavaScript-разработчикам. Мы урезаем десктоп-проекты на JavaScript, пишем теперь только на WPF. И таким образом мы ломаем принцип Interface Segregation, потому что наш класс не может не выполнять свой функционал, и получается что JavaScript разработчики все еще пишут десктоп-проекты.

      Решением этой проблемы будет опять же разделение интерфейса на несколько: IDesktop и IBackend. Это напоминает предыдущий пример, но здесь мы решаем другую проблему — не даем классу делать больше, чем нужно.

    • И теперь последний и, пожалуй, самый трудный для понимания принцип — инверсия зависимостей.

      1. Классы высокого уровня не должны зависеть от классов низкого уровня, при этом оба должны зависеть от абстракций.
      2. Абстракции не должны зависеть от деталей, но детали должны зависеть от абстракций.

      Что это значит? А это значит, что классы высокого уровня реализуют бизнес-правила или логику в системе. Низкоуровневые классы занимаются более подробными операциями, такими как работа с базой данных, передача сообщений в операционную систему - и так далее. Чтобы достичь инверсии зависимостей, мы должны держать эти высокоуровневые и низкоуровневые классы настолько слабосвязанными, насколько возможно. И как раз для этого мы пишем их зависимыми от абстракций, а не друг от друга.

      😭

      class Email {
  public void Send() {
    // Code to send email-letter
  }
}

class Notification {
  private Email email;
  public Notification() {
    email = new Email();
  }

  public void EmailDistribution() {
    email.Send();
  }
}

      🤩

      interface IMessenger {
  void Send();
}

class Email: IMessenger {
  public void Send() {
    /* Code to send email-letter */
  }
}

class SMS: IMessenger {
  public void Send() {
    /* Code to send SMS */
  }
}

class Notification {
  public IMessenger Messenger { get; set; }

  public Notification(IMessenger mess) {
    Messenger = mess;
  }

  public void Notify() {
    Messenger.Send();
  }
}

static void Main(string[] args) {
  var notification = new Notification(new Email());
  notification.Notify(); // Sent email
  notification.Messenger = new SMS(); // Change the provider
  notification.Notify(); // Sent SMS
}

      Давайте рассмотрим этот принцип на примере рассылки сообщений. На первом примере кода класс Notification полностью зависит от класса Email, потому что он отправляет только один тип сообщений. Что если мы захотим отправлять каким-то другим способом? Тогда нам надо будет копаться во всей системе сообщений. Это является признаком того, что система слишком тесно связана.

      Чтобы сделать ее слабо связанной в этом случае нам нужно абстрагироваться от провайдера отправки сообщений Email. Для этого мы создаем интерфейс IMessenger с методом Send и реализуем его в двух классах ― Email и SMS. Класс Notification мы составляем так, чтобы отвязаться от конкретной реализации рассылки сообщений. В этом случае мы можем использовать принцип Dependency Injection, пробросив объект Messenger через конструктор. И в результате мы будем отправлять сообщения того класса, с которым сейчас работаем. Если мы создаем Notification с Email Messenger-ом, отправляется электронное письмо. Далее мы захотели сменить провайдера и присвоили свойства Messenger классу SMS, поэтому следующий вызов метода Notify уже пришлет SMS-ку.

    Каждый принцип SOLID предлагает нам способ написания логического, надежного и понятного кода, а язык С#, при соблюдении этих принципов, позволяет писать большие программы и легко их расширять.
